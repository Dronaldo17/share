推演包依赖问题的产生的原因

单组应用

开始一个新的项目，基本上是这样的。第一个人建立起基本的项目模板：

npm install react-native-cli -g
react-native init AwesomeProject
npm shrinkwrap
git push

其他开发者：

npm install react-native-cli -g
git clone
npm install

react-native init AwesomeProject这一步，约等于

mkdir AwesomeProject && cd AwesomeProject
npm install react-native --save
copy模板（index.ios.js等）

npm shrinkwrap这一步在项目根目录下产生了一个npm-shrinkwrap.json文件，这个文件锁定了整个项目依赖的第三方组件版本。因为npm上众多的源采用Semantic的版本标准，不可避免的出现这样的情况

A@0.0.1 -> ^B@0.0.1

项目中运用了A@0.0.1的模块，依赖树可能是这样的

project -> A@0.0.1 -> B@0.0.1

当B模块发布了0.0.2版本后，其他开发者重新clone代码仓库并且npm install之后，依赖树就变成

project -> A@0.0.1 -> B@0.0.2

npm-shrinkwrap.json则可以避免这一问题的产生，因为npm install命令在构建依赖树时，package.json和npm-shrinkwrap.json文件都会产生影响，npm-shrinkwrap.json的优先级更高。

shrinkwrap这一步非常重要，他保证了项目依赖树不会随着时间推移而产生变化。它保证开发、测试和上线这样一段时间内，每次构建的时候，产生的内容一致。

挖一挖npm publish和npm install命令，看看npm-shrinkwrap.json怎么影响install过程

npm publish命令

众所周知，npm publish是将模块发布到npm源的命令。publish过程是什么？我们找个模块来看看。

看来publish主要分为两步：

  1. 将代码打成tgz包，并放在npm的cache目录下。在这一步，.npm-ignore文件和package.json内的files字段配置都可以配置tgz所包含的内容。在不做任何配置的情况下，npm-shrinkwrap.json会被压进tgz包。
  2. 上传tgz并更新npm源。

npm install过程解析

开始这一节之前，我们先约定一个模块的“简洁”程度。一个模块在分发后，不需要安装其他任何模块即可正常工作，就称为“简洁”模块，例如jQuery@2.*；否则，称为“非简洁”模块，例如react-native。注意这不是通用的叫法，只适用于本次的谈论。在写下这篇讨论时，jquery最新版为2.2.4。

npm install一个“简洁”模块

我们运行调高npm的日志等级，使用npm install jQuery --save --verbose，简单看看npm install命令会发生什么事情。

可以看到，对最简单的一个模块的install过程，主要分为3步

  1. 从https://registry/jquery上获取安装包的信息，包含了版本历史、依赖、作者和各个版本的tgz下载链接等等。
  2. 下载第一步的结果得知的tgz链接，解压缩jquery.tgz包，读取package.json的dependencies。
  3. 重新解压缩jquery.tgz包到node_modules。

非常简单，从日志上就可以对过程有一个感性的认知。那么npm install一个“非简洁”模块会怎样？

给“非简洁”模块加入npm-shrinkwrap.json

我们给简单的给zilong-a模块升级一下，先将package.json内的version修改为0.0.2，然后将一个较低版本的jquery放入zilong-a的依赖树：

qnpm install jquery@2.2.0 --save

qnpm shrinkwrap

qnpm publish

检验一下服源上的tgz的内容，npm-shrinkwrap.json已经被压入了压缩包。

npm install @qnpm/zilong-a@0.0.2 --save --loglevel=silly，检查对比安装0.0.1版本的日志，发现install过程产生了变化：

  1. 第二步中不仅读取了package.json，还多了一步addShrinkwrap
  2. 原来第三步中解析jquery版本的步骤被越过，直接从源上下载了2.2.0版jquery

这里依然符合我们对shrinkwrap命令的认知，即shrinkwrap可以锁定版本依赖树。梳理zilong-a@0.0.2一下目前的依赖结构，目前只有两级：

@qnpm/zilong-a@0.0.2 > jquery@2.2.0

整个下载过程是

zilong-a > jquery

继续将“非简洁”模块复杂化

我们继续发布@qnpm/zilong-a@0.0.3，使其依赖树加深。首先发布一个@qnpm/zilong-b@0.0.1模块，@qnpm/zilong-b依赖jquery@*;去掉@qnpm/zilong-a的jquery依赖，加入@qnpm/zilong-b@*。那么依赖结构变化为：

@qnpm/zilong-a@0.0.3 > @qnpm/zilong-b@0.0.1 > jquery@*

清理zilong-a模块并重新npm shrinkwrap，并发布

rm -rf node_modules && qnpm install qnpm shrinkwrap qnpm publish

zilong-a@0.0.3模块依赖被锁定为 @qnpm/zilong-b@0.0.1和最新的jquery@2.2.4。那么install @qnpm/zilong-a@0.0.3会怎样？

我们发现在安装@qnpm/zilong-a@0.0.1时理解的第三步，又发生了变化。原本应该是

下载zilong-a@0.0.3.tgz > 解析package.json > 下载zilong-b@0.0.1 > 解析package.json > 下载jquery@2.2.4

变成了

下载zilong-a@0.0.3.tgz > 解析package.json > 下载zilong-b@0.0.1并且下载jquery@2.2.4

可见shrinkwrap不仅仅影响了npm的版本判断，而且改变了下载流程，使得下载过程变成了并行的。 官方在npm的github的讨论中声明：对于下载请求，没有任何的队列或者限制机制，有多少下载，就并发多少。参看issue11117和 issue11125，这两个问题都是因为第三方源，限制了下载请求而导致。 这就是下载时tgz文件损坏的罪魁。即对于超级大库而言，庞大的并发请求数量造成网络负载过重；在采用不经优化的网络（比如墙内访问墙外），文件坏掉的可能性增加。shrinkwrap加剧了这个问题。

小结

我们先看一下npm官方和react-native官方两方对于shrinkwrap的态度，理一理可能的方向。

####### react-native对于shrinkwrap的态度 对于使用shrinkwrap的态度。可以发现，在github的提交记录中，直到0.22.0-rc2之前，rn的根目录下都有npm-shrinkwrap.json；但是非常有意思的是，从npm源直接下载的tgz的文件内不包含这个文件。所以我认为，在0.22.0之前，官版对于锁定版本的态度是模糊的，虽然社区内有这样的呼声。直到0.22.0-rc2，官方锁定了依赖版本，然后社区迅速的反应了安装问题。大约一个钟头后，官方不仅从发布的脚本内删除了npm shrinkwrap，并且坚决的从github的目录内删除了npm-shrinkwrap.json文件。fb的一位开发者事后在自己的facebook上回顾这个事件并写下：

... This（就是npm-shrinkwrap.json文件） was added in one of the -rc releases, so finding it was fairly easy. I can't be 100% certain what was the reason, npm client is a big box of wonders.

直到今天官版也没有再将这个文件加回来，可见友谊的小船已翻了个底儿朝天。

npm对于shrinkwrap的态度

npm大约在3.7.x前后（2016年年初）修复了一版shrinkwrap的平台依赖问题，他们的策略是“对于npm-shrinkwrap.json内的平台不兼容的模块，打出一个[warning]或者[info]及的log，然后继续安装。”这个方式感觉不是那么完美，但是对于大部分模块来说，似乎已经足够了（仍然有少量ubuntu用户在3.8.x报错）。

综合来看，即使npm官方在3.7.x版本修复了shrinkwrap，考虑到众多低版本npm的使用者，react-native短期内也不可能再加入shrinkwrap。

只有自己动手了。我们缺的，只是一个能够区分os的npm-shrinkwrap.json。
